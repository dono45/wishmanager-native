# 小管家 - 儿童财商与愿望管理工具

> 帮助 3-12 岁儿童建立健康的金钱观和消费习惯的移动端应用。
> 完全离线运行，数据存储在本地 SQLite 数据库中。

---

## 一、产品设计愿景

### 1.1 背景

现代消费社会中，儿童过早接触到"即时满足"的消费模式，容易形成冲动消费的习惯。家长在孩子要买玩具时面临两难：无条件满足会纵容消费，直接拒绝又可能伤害亲子关系。

### 1.2 核心价值

**小管家**通过"7天冷静期"核心机制，帮助孩子和家长建立健康的消费决策流程：

1. **延迟满足训练** - 孩子需要经过7天思考才能购买想要的物品
2. **可视化进度** - 环形进度图、确认天数、剩余时间等视觉反馈让等待过程有成就感
3. **预算意识培养** - 实时检查预算，超预算物品直接标记为"预算不足"
4. **家长决策支持** - 急救包提供标准化的冲突应对方案
5. **游戏化激励** - 星星系统让孩子主动参与财商培养

### 1.3 设计原则

- **离线优先** - 所有数据本地存储，无需网络
- **儿童友好** - 童趣表情、暖色调、鼓励弹框
- **家长可控** - 独立家长密码，预算设定、消费分析、紧急应对
- **渐进式教育** - 从冷静期到预算管理，逐步建立财商意识
- **单物品决策** - 每个愿望独立判断，互不影响

## 五、核心机制说明

### 5.1 冷静期状态机

```
                    (添加愿望 + 预算检查)
                              ↓
                    ┌─────────────────┐
                    │  budget check   │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              ↓                             ↓
        [insufficient]                [cooling] ← 7天冷静期
        (预算不足)                         │
              ↑                            │
              │              ┌─────────────┼─────────────┐
              │              ↓             ↓             ↓
        (restore)      (不想要了)   (第7天未确认)   (第7天已确认)
              │              ↓             ↓             ↓
              │        [cancelled]    [expired]     [wanted]
              │                                         │
              │                          ┌──────────────┼──────────────┐
              │                          ↓              ↓              ↓
              │                     (预算不足)      (手动取消)     (预算充足)
              │                          ↓              ↓              ↓
              └──────────────── [insufficient]   [cancelled]   [purchased]
```

### 5.2 预算检查时机

| 时机 | 行为 |
|------|------|
| **创建愿望时** | 立即检查当月预算，超预算 → `insufficient`，不进入冷静期 |
| **恢复冷静期时** | `insufficient` 点击"我还想要"，重新检查预算，够则重置7天冷静期 |
| **冷静期结束点击购买时** | `purchaseSingleWish(id)` 只处理当前这一个愿望，不影响其他 |
| **页面挂载时** | `processCooling()` 批量处理所有已过期且未确认的冷静期愿望 |

### 5.3 预算滚存逻辑

```
每月1日:
  当月总预算 = 基础月预算 + max(0, 上月剩余)
  当月剩余 = 当月总预算
  当月已花 = 0

购买时:
  剩余 -= 物品价格
  已花 += 物品价格
```

### 5.4 单物品购买（purchaseSingleWish）

```
输入: wishId

1. 查询该愿望，确认 status === 'cooling'
2. 查询当月预算 remaining
3. 如果 remaining >= wish.price:
     → 扣减预算
     → status = 'purchased'
     → 返回 { status: 'purchased', message: '太棒了！xx 已购买！' }
   否则:
     → status = 'insufficient'
     → 返回 { status: 'insufficient', message: 'xx 预算不够，下月再来吧~' }

⚠️ 只处理这一个愿望，不检查其他愿望的确认记录
```

### 5.5 批量过期处理（processCooling）

```
1. 查询所有 cooling_end_at <= 当前时间的愿望
2. 对每个愿望:
   - 如果 dailyConfirmations 不包含最后一天 → status = 'expired'
   - 否则检查预算 → purchased / insufficient

⚠️ 仅在页面挂载时自动调用，不响应用户点击
```

## 六、部署方式

### 6.1 环境要求

- Node.js >= 18
- npm 或 yarn
- Android Studio（Android开发）或 Xcode（iOS开发）

### 6.2 开发环境搭建

```bash
# 1. 进入项目目录
cd wishmanager-native

# 2. 安装依赖
yarn install

# 3. 启动 Expo 开发工具（Metro bundler，仅开发阶段使用）
npx expo start

# 4. 连接真机或模拟器运行
#   方式A：按 a 键直接在已连接的 Android 设备上运行
#   方式B：在 Android Studio 中打开 android/ 目录构建
npx expo run:android
```

> **注意**：`npx expo start` 仅在开发阶段用于代码打包和热更新。最终构建的 APK 完全离线运行，不依赖任何网络连接。

### 6.3 构建 APK（Android）

```bash
# 使用 EAS Build（推荐）
npm install -g eas-cli
eas login
eas build:android --profile preview

# 或本地构建（需 Android SDK）
cd android
./gradlew assembleRelease
```

### 6.4 调试面板（时间模拟）

儿童模式「我的」Tab 底部有一个**调试时间面板**，用于开发阶段快速模拟时间流逝，测试冷静期、月度预算等时间相关逻辑：

```tsx
// src/screens/KidHome.tsx
{true && <DevPanel />}
```

| 按钮 | 功能 |
|------|------|
| +1天 | 快进 1 天 |
| +7天 | 快进 7 天 |
| +30天 | 快进 30 天 |
| 重置 | 恢复真实时间 |

> **注意**：发布前应将条件改回 `{__DEV__ && <DevPanel />}`，确保正式包不显示调试面板。

### 6.5 数据存储位置

SQLite 数据库文件存储在应用私有目录中，随应用卸载而清除。

## 十、技术选型说明

### 为什么用 React Native + Expo 而不是原生 Android？

1. **业务逻辑复用** - 原项目是 React + TypeScript，组件结构、状态管理逻辑可最大程度复用
2. **跨平台** - 同一套代码支持 Android 和 iOS
3. **开发效率** - Hot Reload 大幅提升开发速度
4. **原生性能** - Expo 模块（SQLite/SecureStore/ImagePicker）均为原生实现，无性能损失
5. **离线能力** - SQLite 提供完整的本地数据库能力，无需网络

### 为什么不实现云同步？

按需求约定，本项目为纯本地应用，数据不离开设备。如需未来扩展云同步，可在已有 SQLite 基础上增加服务端同步层，不影响现有架构。
